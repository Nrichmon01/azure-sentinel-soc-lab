## Executive summary

Microsoft Sentinel detected a high volume of sign-in events from a single IP address (`154.161.xxx.xxx`, Location: GH — Ghana), totalling 301 events. This triggered the **Brute force attack against Azure Portal** analytics rule (MITRE T1110, Medium severity).

Investigation confirmed the activity originated from the analyst's own IP during lab setup and testing. No unauthorised access occurred. Incident classified as **Benign Positive** and closed.

## Investigation

### Step 1 — Scope the activity

Query used to group sign-in events by IP:

```kql
SigninLogs
| where TimeGenerated > ago(1h)
| where ResultType != "0"
| summarize FailedAttempts = count(), Users = make_set(UserPrincipalName)
    by IPAddress, bin(TimeGenerated, 5m)
| where FailedAttempts > 10
| order by FailedAttempts desc
```

**Result:** 301 events from a single IP. Activity included both failed and successful sign-ins by the analyst account during lab setup and connector testing.

### Step 2 — IP enrichment

| Field | Value |
|---|---|
| Geolocation | Ghana (GH) — consistent with analyst's location in Accra |
| Threat intelligence | No malicious indicators — not listed on known threat feeds |
| ASN | Confirmed residential/ISP-assigned |

### Step 3 — Check for successful sign-in

```kql
SigninLogs
| where IPAddress == "154.161.xxx.xxx"
| where ResultType == "0"
| project TimeGenerated, UserPrincipalName, IPAddress, Location, AppDisplayName
```

**Result:** Successful sign-ins present, all performed by the analyst account during lab configuration. No post-authentication suspicious activity observed.

### Step 4 — Context assessment

All sign-in events were generated by:

1. The analyst logging into the Azure portal during lab setup
2. Connector configuration requiring repeated authentication
3. Test user `testsocuser` creation and deletion
4. KQL query testing against the live SigninLogs table

No indicators of compromise identified.
